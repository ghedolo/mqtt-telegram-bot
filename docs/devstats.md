# Development-effort stats — how they're computed

The "Development effort" section of the [README](../README.md) is generated by
`devstats.py` from Claude Code session transcripts. This documents what each
number means and how the calculation stays correct across machines.

## The ledger

`devstats.py` does **not** recompute the section from scratch each run — that
would lose work done on other machines and any pruned transcripts. Instead it
keeps a committed ledger, **`devstats.json`**, with one entry per session keyed
by the Claude Code **session UUID** (the transcript's `.jsonl` filename):

```json
{
  "baseline": { "...frozen pre-ledger totals..." },
  "sessions": {
    "<uuid>": { "first": "...", "last": "...", "active_secs": 0,
                "user": 0, "asst": 0,
                "input": 0, "output": 0, "cache_read": 0, "cache_write": 0 }
  }
}
```

Each run scans this machine's transcripts and writes each local session's entry
(overwrite — so re-running just refreshes, never double-counts). Entries from
other machines, already in the committed ledger, are left untouched. The README
is then rendered from the whole ledger.

## Cross-machine and cumulative

Because the ledger is committed, effort accumulates across machines:

1. On machine A: `python3 devstats.py`, then commit + push `devstats.json`.
2. On machine B: `git pull`, work, `python3 devstats.py` (adds B's sessions), commit + push.

A session recorded once survives later transcript pruning — its entry stays in
the ledger even after Claude Code deletes the `.jsonl`.

The transcript directory is derived from the script's own path
(`~/.claude/projects/<project-path-with-'-'-separators>`), so it resolves on each
machine without a hardcoded location — that's why `devstats.py` is portable and
committed rather than machine-local.

## The baseline

History before the `cutoff` (**2026-07-12**) predates the ledger; those
transcripts were pruned and can't be recomputed. Their totals (9 sessions) are
frozen in the `baseline` entry. A session whose first message is **after** the
cutoff accumulates on top; one dated on or before the cutoff is skipped (already
inside the baseline), so nothing is double-counted.

## What each metric means

- **Sessions** — baseline count + number of ledger entries (one per transcript).
- **Messages** — count of `user` and `assistant` records in the transcripts.
- **Active conversation time** — sum of gaps between consecutive timestamps that
  are ≤ 5 min, within each session. Longer gaps (idle, overnight, machine
  switches) are discarded, so it approximates hands-on time, not wall-clock.
- **Tokens** — summed from each message's `usage`:
  - *Input (non-cache)* — `input_tokens`
  - *Output* — `output_tokens`
  - *Cache write* — `cache_creation_input_tokens`
  - *Cache read* — `cache_read_input_tokens`

  The total is dominated by cache reads: the growing conversation is re-read on
  every turn.
- **Output tokens / message** — total output ÷ assistant messages. Early sessions
  used caveman mode (terser output); later prose-heavy sessions raise the average.

## Running it

```bash
python3 devstats.py
git add devstats.json README.md && git commit -m "chore: devstats" && git push
```

Re-running mid-session is safe — the current session's entry is refreshed in
place. Run it at session close for the most complete numbers.
